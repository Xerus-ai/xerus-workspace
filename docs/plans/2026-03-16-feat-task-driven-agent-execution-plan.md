# Task-Driven Agent Execution

## Overview

Make AI agents follow workflow order by making **beads tasks the single source of truth** for what agents do. Instead of relying on agents reading and following CLAUDE.md instructions (which degrades at >60% context usage), enforce workflow order through:

1. **session-start.sh** injects the agent's current task on wake (the agent doesn't decide what to do -- the board tells it)
2. **Shift templates** define recurring daily workflows per channel (research -> draft -> post -> measure)
3. **Shift instantiation** creates today's tasks from the template with proper dependencies
4. **Task completion validation** checks acceptance criteria before allowing `bd close`
5. **metadata_sync bridge** ensures `bd close` in sandbox updates the backend DB and frontend Kanban

## Problem Statement

Agents have instructions (CLAUDE.md, skills, SOPs) but nothing enforces execution order. In the first twitter shift, the research step was skipped entirely -- 15 tweets posted without data backing because no mechanism blocked the content agents from starting before the research agent finished.

The infrastructure exists (beads, dual-write, metadata_sync, Kanban frontend) but the workspace hooks don't activate it. The session-start hook creates empty files instead of injecting tasks. The task-completed hook logs an event instead of validating output.

## Architecture Decisions

### AD1: Two-level .beads/ hierarchy (company + channel)

**Decision**: Two `.beads/` locations serve different scopes:
- **Root `.beads/`** = company-level tasks (cross-channel epics, strategic initiatives, org-wide)
- **Per-channel `projects/{domain}/channels/{channel}/.beads/`** = team-level tasks (daily shift work, channel-specific)

**Why**: Channels are teams with different cadences. Twitter does daily content shifts. A dev channel does sprint tasks. An infra channel does one-offs. Per-channel beads keeps team boards separate. Root beads is for xerus-master's company-level orchestration.

**Impact**: `session-start.sh` reads the agent's **channel-level** `.beads/`. `instantiate-shift.sh` creates tasks in the **channel-level** `.beads/` (runs from the channel directory so `bd` auto-discovers the right DB). xerus-master reads **root** `.beads/` for company tasks. `platform.create_task` MCP handler already writes to per-channel `.beads/` -- this is correct.

### AD2: PreToolUse validation, not post-close reopening

**Decision**: Validate acceptance criteria BEFORE `bd close` executes, not after.

**Why**: If `bd close` already ran and validation fails, you'd need `bd reopen` which is fragile. The SDK's PreToolUse hook can inspect Bash commands and return `{ decision: 'block', message: 'reason' }` to prevent execution. Clean and reversible.

**Impact**: `pre-tool-use.sh` intercepts Bash commands containing `bd close`, reads the task's acceptance criteria, checks deliverable exists, blocks if validation fails.

### AD3: Backend-side PostToolUse hook detects bd commands directly

**Decision**: The backend TypeScript PostToolUse hook (`post-tool-use.hook.ts`) inspects Bash tool calls for `bd close` / `bd update --status` commands. When detected, it reads `.beads/issues.jsonl` from the sandbox via `DaytonaProvider.readFile()` and calls `MetadataSyncService.syncTask()` directly.

**Why**: The backend already receives every tool use event from the runner via stdout. The PostToolUse event includes tool name and input (the bash command string). No intermediate files, no polling, no shell-to-stdout bridging needed. Direct sync on the server side.

**Alternative considered**: Shell-side post-tool-use.sh writing to `.xerus/pending-task-sync.jsonl` for the runner to poll. Rejected because it adds an intermediate file and polling complexity when the backend already has the information it needs.

### AD4: Status mapping: beads -> DB

| Beads Status | DB Status | Notes |
|-------------|-----------|-------|
| `open` | `todo` | Existing mapping |
| `in_progress` | `in_progress` | Direct match |
| `closed` | `done` | **NEW -- must add** |
| `blocked` | `todo` | With `metadata.blocked: true` |

### AD5: .task-context.md location

**Decision**: Write to `.memory/agents/{slug}/.task-context.md` (agent-scoped, survives compaction because it's on disk).

**Why**: `context/trigger/` is for backend-generated trigger files. `.task-context.md` is generated by the sandbox-side hook, so it belongs in the agent's memory space. The agent reads `.memory/agents/{slug}/working.md` already -- `.task-context.md` sits next to it.

### AD6: Template repo = infrastructure only, instance = user content

**Decision**: xerus-workspace (template) contains:
- `instantiate-shift.sh` script (generic, works with any channel's shift.yaml)
- `shared/office/shift.yaml.example` (schema reference)
- Updated hooks (session-start.sh, pre-tool-use.sh)
- Updated CLAUDE.md, OPERATING.md

workspace-light (and any user sandbox) contains:
- Actual `shift.yaml` files per channel (user-specific agents, tasks, deliverables)
- Channel-level `.beads/` with real tasks

**Why**: The template ships to every user. It should have tools and infrastructure, not content. Shift templates are created when users set up their channels and hire agents. workspace-light is our test bed to verify the template works before it ships.

## Technical Approach

### Phase 1: Foundation (Backend + Workspace Template)

Fix the plumbing so tasks flow correctly between sandbox and backend.

#### Task 1.1: Fix status mapping in MetadataSyncService

**File**: `glass/xerus_backend/src/domains/execution/metadata-sync/metadata-sync.service.ts`

Add missing status mappings:
```typescript
function mapBeadsStatusToDb(beadsStatus: string): string {
    const mapping: Record<string, string> = {
        'open': 'todo',
        'in_progress': 'in_progress',
        'closed': 'done',
        'blocked': 'todo',  // with metadata flag
    };
    return mapping[beadsStatus] || 'todo';
}
```

**Acceptance criteria**:
- [ ] `closed` maps to `done`
- [ ] `blocked` maps to `todo` with metadata
- [ ] Existing `open -> todo` mapping preserved
- [ ] No DB CHECK constraint violations

#### Task 1.2: Add bd-command detection to backend PostToolUse hook

**File**: `glass/xerus_backend/src/domains/execution/hooks/post-tool-use.hook.ts`

The backend already receives PostToolUse events with tool name + input. When tool is "Bash" and command contains `bd close` or `bd update --status`:

1. Parse task ID from command string (regex: `bd\s+(close|update)\s+([^\s]+)`)
2. Determine the channel-level `.beads/issues.jsonl` path (from agent's channel assignment)
3. Read the file from sandbox via `DaytonaProvider.readFile()`
4. Find the task entry by ID, extract current status
5. Call `MetadataSyncService.syncTask()` with mapped status (`closed` -> `done`)

**Acceptance criteria**:
- [ ] Detects `bd close <id>` in Bash command string
- [ ] Detects `bd update <id> --status <status>` in Bash command string
- [ ] Reads task data from sandbox filesystem (not local)
- [ ] Calls MetadataSyncService.syncTask() with correct status mapping
- [ ] Does not interfere with non-bd Bash commands
- [ ] Handles cases where .beads/ doesn't exist (skip gracefully)

### Phase 2: Session Start Enhancement

#### Task 2.1: Enhance session-start.sh to inject task context

**File**: `xerus-workspace/.claude/hooks/scripts/session-start.sh`

After existing setup (mkdir, init-db, working.md), add:

1. Resolve agent's channel from `agents/{slug}/CLAUDE.md` (grep for `Primary:` line)
2. Read channel-level `.beads/issues.jsonl` from that channel directory (or use `cd {channel_dir} && bd list --json -a $AGENT_SLUG --status open`)
2. For each assigned task:
   - Check dependency status via `bd show <id> --json` (look at deps)
   - Classify as READY (all deps closed) or BLOCKED (any dep open)
3. Write `.memory/agents/{slug}/.task-context.md`:

```markdown
# Task Context -- {agent_slug}
Generated: {timestamp}

## Current Task (READY)
**ID**: workspace-light-3t5.2
**Title**: Draft 3 tweets from research brief
**Priority**: P2
**Acceptance**: output/deliverables/tweets-2026-03-16.md must exist with >200 bytes

### Description
Read content-lab research brief. Draft 3 tweets: 2 dogfooding proof + 1 viral hot take.

### Required Skills
- twitter-engagement
- bird

### Dependencies (all resolved)
- workspace-light-3t5.1: Research trending topics (CLOSED by curator-carla)

## Blocked Tasks
- workspace-light-3t5.5: Post all content (waiting on: 3t5.3, 3t5.4)

## Instructions
1. Do the Current Task above. Nothing else.
2. When done, close it: `bd close workspace-light-3t5.2 --reason "your summary"`
3. Save progress to .memory/agents/{slug}/working.md
```

4. If ALL tasks are BLOCKED, write:
```markdown
# Task Context -- {agent_slug}
Generated: {timestamp}

## Status: BLOCKED
All assigned tasks have unresolved dependencies.

### Blocked Tasks
- workspace-light-3t5.2: Draft tweets (waiting on: 3t5.1 - Research)

## Instructions
No work to do. Output "BLOCKED: waiting for [dependency list]" and end session.
```

**Acceptance criteria**:
- [ ] Resolves agent's channel directory from CLAUDE.md
- [ ] Reads from channel-level `.beads/issues.jsonl` (or `bd list --json` from channel dir)
- [ ] Filters tasks by `$XERUS_AGENT_SLUG` in assigned_agents
- [ ] Checks dependency status for each task
- [ ] Picks highest priority READY task as "Current Task"
- [ ] Writes `.task-context.md` to `.memory/agents/{slug}/`
- [ ] Handles: no tasks assigned, all blocked, multiple ready (picks highest priority)
- [ ] Handles: beads not initialized (no .beads/ dir) -- skip gracefully

### Phase 3: Shift Templates + Instantiation

#### Task 3.1: Write instantiate-shift.sh script

**File**: `xerus-workspace/.claude/hooks/scripts/instantiate-shift.sh`

Script that:
1. Accepts channel path as argument: `./instantiate-shift.sh projects/xerus-launch/channels/twitter`
2. Reads `{channel}/shift-template.md` (or .yaml)
3. Checks idempotency: searches for existing epic with today's date via `bd list --labels "shift-{date}" --status open`
4. If no existing epic:
   a. Creates epic: `bd create "Twitter Shift -- {date}" -t epic -p 1 -l "shift-{date}"`
   b. For each task in template:
      - `bd create "{title}" -a {assignee} -p {priority} --parent {epic_id} -l "shift-{date}"`
   c. For each dependency:
      - `bd dep add {child_id} {parent_id}`
5. Outputs created task IDs

**Acceptance criteria**:
- [ ] Parses shift-template.md YAML correctly
- [ ] Creates epic + child tasks with proper hierarchy
- [ ] Sets dependencies between tasks
- [ ] Assigns agents via `-a` flag
- [ ] Idempotent: running twice on same day does not create duplicates
- [ ] Substitutes `{date}` placeholder in acceptance criteria paths
- [ ] Outputs task IDs for verification
- [ ] Works with `bd` CLI v0.42.0+

### Phase 4: Task Completion Validation

#### Task 4.1: Add bd-close interception to pre-tool-use.sh

**File**: `xerus-workspace/.claude/hooks/scripts/pre-tool-use.sh`

When a Bash command contains `bd close <task-id>`:
1. Parse the task ID
2. Read the task from `.beads/issues.jsonl` (find by ID)
3. Look for `acceptance` in the task's metadata (or from shift template)
4. If acceptance criteria specify a deliverable path:
   - Check if file exists at the specified path
   - Check if file size meets minimum (if specified)
5. If validation fails: output error message with what's missing, exit with non-zero code (blocks the tool)
6. If validation passes: exit 0 (allows bd close to proceed)

**Acceptance criteria**:
- [ ] Detects `bd close` in Bash command string
- [ ] Parses task ID from command
- [ ] Reads acceptance criteria from task metadata
- [ ] Checks deliverable file existence and size
- [ ] Blocks with clear error message on failure
- [ ] Passes silently on success
- [ ] Does not interfere with other Bash commands
- [ ] Handles tasks without acceptance criteria (passes by default)

#### Task 4.2: Store acceptance criteria in beads task metadata

When `instantiate-shift.sh` creates tasks, it needs to store acceptance criteria in the task metadata so `pre-tool-use.sh` can read them. Use `bd create` with `--acceptance` flag:
```bash
bd create "Draft 3 tweets" -a thread-theo --acceptance "output/deliverables/tweets-2026-03-16.md exists and > 200 bytes"
```

Or store in task description if --acceptance is not supported.

**Acceptance criteria**:
- [ ] Acceptance criteria stored in beads task (metadata or description)
- [ ] pre-tool-use.sh can read them from `bd show` or issues.jsonl
